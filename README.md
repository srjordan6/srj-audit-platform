# SRJ AI Audit Platform

Tier 1/2/3 AI governance audit platform for SRJ Consulting & Services LLC.

**Domain:** [aiauditforcompanies.com](https://aiauditforcompanies.com)
**Spec:** Part A v1.2 (rendered from Part A v1.1 + Part A v1.2 Patch Specification, 2026-06-23)
**Tier 1 question count:** 115 (loaded into the `questions` PostgreSQL table on 2026-06-23)

## Architecture

Single Django monolith deployed to Render as a Web Service, backed by Render-managed
PostgreSQL 18 and Redis. Background work (PDF generation, email delivery, score
calculation) runs in a separate Render Background Worker. File storage on Backblaze B2
via the S3 API. Payments via Stripe. Email via Postmark.

A separate read-mostly MCP server (`srj-mcp`, deployed to Render) exposes the same
PostgreSQL database for AI-agent operator access. The MCP server uses
`srj_audit_app` (NOBYPASSRLS) and binds tenant via `SET LOCAL app.current_tenant_id`
per request. Django uses the same role.

## Local setup

```bash
# Prereqs: Python 3.12, Docker (for local Postgres + Redis)

# Local Postgres
docker run --name srj-postgres -e POSTGRES_PASSWORD=devpass -p 5432:5432 -d postgres:16
docker exec srj-postgres createdb -U postgres srj_audit

# Local Redis
docker run --name srj-redis -p 6379:6379 -d redis:7

# Python environment
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Copy environment template and fill in values
cp .env.example .env
# Edit .env — at minimum set SECRET_KEY, DATABASE_URL, REDIS_URL

# First run — fake-applies the schema that srj-mcp bootstrapped, then runs forward
python manage.py migrate --fake-initial
python manage.py createsuperuser
python manage.py runserver

# Visit:
#   http://localhost:8000/         — service status JSON
#   http://localhost:8000/admin/   — Django admin
#   http://localhost:8000/healthz/ — health probe
```

## Schema ownership

The PostgreSQL schema was bootstrapped by the `srj-mcp` server (2026-06-23). All
12 tables (companies, users, engagements, respondents, questions, responses,
scores, reports, documents, events, discount_credits, refund_requests) plus
indexes, RLS policies, and grants exist before Django runs.

Django models in this codebase use `Meta: managed = False` so initial migrations
do not attempt to recreate those tables. On first deployment, run
`python manage.py migrate --fake-initial` to register the existing schema as
Django-tracked. From that point, schema evolutions go through Django migrations
(generated by `makemigrations`, applied normally by `migrate`).

If a column needs to be added (e.g., AbstractBaseUser fields that don't exist
yet on the users table), generate a Django migration that runs an `ALTER TABLE`
statement, then flip `managed = True` on the relevant model.

## Deployment to Render

1. Push this repository to GitHub.
2. In the Render dashboard, create a new Blueprint Instance and point it at
   the repository. The included `render.yaml` provisions the web service and
   background worker.
3. Set environment variables in the Render dashboard (do NOT commit `.env`):
   - `SECRET_KEY` (generate with `python -c "import secrets; print(secrets.token_urlsafe(50))"`)
   - `DATABASE_URL` — paste the internal connection string from the existing
     `srj-audit-db-prod` Postgres instance
   - `REDIS_URL` — paste the internal connection string from `srj-audit-redis-prod`
   - `ALLOWED_HOSTS` — `aiauditforcompanies.com,srj-audit-web.onrender.com`
   - Stripe, Postmark, Backblaze B2, Sentry credentials per `.env.example`
4. First deploy will run `pip install`, `collectstatic`, and `migrate --fake-initial`.
5. Add custom domain `aiauditforcompanies.com` in Render → Settings → Custom Domains.
6. In Cloudflare, add the A or CNAME record Render provides (proxied, orange cloud).
   Set SSL/TLS mode to "Full (strict)".
